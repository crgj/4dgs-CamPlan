# AI Coder 信息交换协议（Planner）

> 本文件是**权威协议**。多个 AI coder 在本项目协作时，**必须**遵守。配套技能：`.agents/skills/coder-handoff/SKILL.md`（人读摘要）、`.agents/skills/planner-conventions/SKILL.md`（编码约定）。每个 coder 第一次接手时先读本文件。

## 0. 目标

让多个 AI coder（可能来自不同模型、不同会话、不同操作者）能**并行/接力**开发 Planner，做到：

- 上下文不丢失（靠日志与状态文件）
- 任务不重复、不互相覆盖（靠看板 + git 分支）
- 决策与坑可追溯（靠 LOG 的 DECISION/BLOCKED 行）
- 交接零摩擦（靠 HANDOFF 检查清单）

## 1. 角色与标识

每个 coder 有一个 **coder 标识**，格式 `<模型或代号>/<操作者>`：

- 例：`glm-5.2/crgj`、`claude-sonnet-4/alice`、`gpt-5/bob`。
- 在所有 LOG 行、TASKS 领取记录、commit message 尾部、HANDOFF 中使用同一标识。
- 单人指挥多个 coder 时，操作者字段相同、模型字段不同即可区分。

## 2. 信息交换文件清单（项目根目录）

| 文件 | 性质 | 写入策略 | 内容 |
|------|------|----------|------|
| `HANDBOOK.md` | 稳定 | 低频整体更新 | 项目总览、目录地图、当前阶段、快速上手 |
| `.coder-protocol.md` | 稳定 | 本文件，几乎不改 | 协议本身 |
| `PLAN.md` | 半稳定 | 里程碑达成时更新 | 总体规划、阶段、里程碑与验收 |
| `TASKS.md` | 高频 | 领取/完成即改 | 任务看板：Backlog / Doing / Done |
| `LOG.md` | 最高频 | 每个有意义的步骤追加 | 共享时间线日志（倒序） |
| `STATE.md` | 高频 | 每次显著进展或交接 | 当前世界状态快照 |
| `docs/architecture.md` | 半稳定 | 架构变更时 | 架构与数据流图 |
| `docs/notes/<module>.md` | 按需 | 该模块有决策/坑时 | 模块级设计笔记 |

**规则：** 只读文件不要改（HANDBOOK/PLAN/架构除非你确实在推进它）；高频文件（LOG/TASKS/STATE）用“追加 + 原子写”。

## 3. LOG.md 行格式（强制，机器可解析）

每行：

```
[<UTC ISO8601 时间戳>] [<coder 标识>] <TAG> <任务id或主题> | <正文> | <可选键值对>
```

- **时间戳**：UTC，`date -u +%Y-%m-%dT%H:%M:%SZ`，方括号包裹。
- **TAG**（固定枚举）：
  - `START` —— 开始一个任务
  - `PROGRESS` —— 关键进度（非每行小步）
  - `DECISION` —— 架构/方案决策（含原因）
  - `BLOCKED` —— 被阻塞，写明等待什么
  - `UNBLOCK` —— 解除阻塞
  - `DONE` —— 任务完成，附测试结果与 commit
  - `HANDOFF` —— 交接，含未完事项与下一步
  - `NOTE` —— 备注（坑、提醒、观察）
- **正文**：人类可读的一句话。
- **键值对**（可选，`|` 分隔，`键: 值`）：如 `commit: a1b2c3d`、`tests: 18 passed`、`files: src/sim/coverage.ts`、`wait-for: task-007`。
- **追加到文件顶部**（最新在上）。
- **不要删行**，只追加；错了就再追加一行 `NOTE` 更正。

示例见 `.agents/skills/coder-handoff/SKILL.md`。

## 4. 任务生命周期（看板 TASKS.md）

每个任务：

```
### T-012 视锥栅格化覆盖计数
- 状态: Done          # Backlog | Doing | Done | Blocked
- 负责: glm-5.2/crgj
- 验收: coverage.ts 对采样点正确判定在/不在视锥；单测 ≥ 8 例通过
- 依赖: T-005
- 产物: src/sim/coverage.ts, src/sim/coverage.test.ts
- commit: a1b2c3d
- 备注: 栅格默认 256，遮挡用 AABB 近似（v2 改光线投射）
```

状态机：`Backlog →(领取)→ Doing →(验收通过)→ Done`；Doing 中遇阻 → `Blocked`（标 `wait-for`）。

**领取动作（防并发）：**
1. 推荐：**每个任务一条 git 分支** `T-012/coverage`，领取即建分支，完成开 PR 合 main。git 天然串行化合并，最稳。
2. 退而求其次（单仓直接改）：领取时用 `LOG.md` 顶部 `START` 行声明所有权；其他人看到 Doing 状态不得重复领取。
3. 改 `TASKS.md`/`STATE.md` 前先 `git pull --rebase`，小步提交，减少冲突。

## 5. 验收门槛（DoD — Definition of Done）

一个任务标 Done 前必须满足：

- [ ] 代码符合 `planner-conventions`（分层、纯函数在 sim/export、类型契约、单位）
- [ ] `npm run typecheck` 通过
- [ ] `npm run test` 通过（新增纯函数必有单测）
- [ ] `npm run lint` 通过
- [ ] `LOG.md` 有 `DONE` 行（含 tests 与 commit）
- [ ] `TASKS.md` 状态更新
- [ ] 若改动 `src/types/` 或共享接口：`LOG.md` 有 `DECISION` 行说明兼容性

## 6. 交接流程（HANDOFF）

结束会话或换人时，**完成全部**：

1. 改动已 commit + push（或分支已开 PR）。
2. `TASKS.md` 状态真实。
3. `LOG.md` 顶部追加 `HANDOFF` 行，至少写：未完事项、下一 coder 起点、注意事项。
4. `STATE.md` 更新到当前真实状态（阶段、可跑命令、已知问题、下一步）。
5. 本地 `typecheck/test` 通过，或注明失败与原因。
6. 有阻塞则 `BLOCKED` 行已注明等待项。

## 7. 冲突与共享区处理

- **`src/types/`（共享类型）**：改动前先在 LOG 发 `DECISION`；优先**追加可选字段**而非改已有字段签名，保持向后兼容。
- **跨模块接口**（如 `sim` ↔ `scene` 经 `SceneDef`）：接口变更属架构改动，走 PLAN 里的“架构稳定”里程碑后才大改。
- **大范围重构**：单独立任务，LOG 广播 `DECISION` + `HANDOFF`，选无其他人改动的时机合并。
- 频繁 `git pull --rebase`；冲突优先沟通（在 LOG 留 `NOTE` @对方）。

## 8. 命名约定

- 任务 id：`T-NNN`（三位数字，按 TASKS.md 顺序）。
- 分支：`T-NNN/<slug>`。
- commit：`T-NNN: <动词短语>`，尾部附 `[coder: <标识>]`。
- 实体/类型：见 `planner-conventions`。

## 9. 工具命令速查（每个 coder 接手先确认可用）

```bash
date -u +%Y-%m-%dT%H:%M:%SZ          # 生成 LOG 时间戳
git pull --rebase                      # 同步
npm run typecheck && npm run test && npm run lint   # 验收三连
npm run dev                            # 本地跑（规划中，见 PLAN）
```

## 10. 协议变更

本协议本身改动需：在 LOG 发 `DECISION` 说明，更新版本号（文件头），并通知所有在 Doing 的 coder 通过 `STATE.md` 顶部提示。
