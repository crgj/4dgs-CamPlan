# Planner — V1 总体规划与执行计划（PLAN.md）

> 版本：v1.0　作者：glm-5.2/crgj　日期：2026-06-19
> 这是 Planner 的**主规划文件**。阶段、里程碑、验收、逐步执行都在此。里程碑达成时更新状态。
> 任务清单见 `TASKS.md`（T-NNN），架构见 `docs/architecture.md`，领域知识见 `.agents/skills/4dgs-capture/SKILL.md`。

## 1. 项目目标（v1 范围）

构建一个 **Web 端 4DGS 拍摄规划仿真器**，让用户：

1. 在 3D 场景中**拖放**摄像机阵列、灯光、被拍主体、环境（HDRI/地面）。
2. 实时**仿真拍摄效果**：视锥可视化、覆盖热图、盲区、重叠率、baseline、曝光一致性。
3. **评估采集质量**：以可配置阈值给出覆盖/盲区/重叠/曝光的量化指标与告警。
4. **生成仿真数据**：导出 Nerfstudio `transforms.json` 与 COLMAP 三件套；生成拍摄清单（位姿/曝光/触发时刻）。
5. 提供**专业工具风 UI**，**严格参考 Unreal Editor 5.8**（深色，顶栏/左 Outliner/中视口/右 Details/底状态条，数值 scrub、向量轴色、gizmo 等），含完整快捷键。

**v1 不做**（明确排除，避免范围蔓延）：真实光线追踪渲染预览、深度/法线 G-Buffer 导出、动态主体时序重建、硬件同步协议、点云导出（COLMAP points3D 留空）。这些进 v2。

**4DGS 预留**：`CameraDef`/`SceneDef` 带可选 `time` 字段；拍摄清单含触发时刻；主体静态但类型不写死，便于 v2 加时序形态。

## 2. 技术决策（锁定）

| 维度 | 选择 | 理由 |
|------|------|------|
| 渲染 | Three.js + R3F + drei | 生态最大，专业 UI 与控件最齐 |
| 状态 | Zustand（单 store + 切片） | 轻量、R3F 友好、易测 |
| UI | Tailwind + CSS 变量，**严格参考 Unreal Editor 5.8** 暗色主题 | 专业标杆；信息密集、控件成熟（见 ue5-ui-reference skill） |
| 构建 | Vite + TS(strict) | 快、类型安全 |
| 测试 | Vitest（纯函数）+ Playwright（E2E，后期） | sim/export 可纯单测 |
| 虚拟列表 | @tanstack/react-virtual | 大纲/统计性能 |

**架构铁律**：`sim/` 与 `export/` 纯 TS，禁 React/Three 依赖；Three 对象只在 `scene/`；状态只走 store；角度用度，换算只在 `lib/math`。

## 3.1 阶段2：P7 高质量渲染（v1 之后，高优先级 Backlog）

> **背景**：原计划的 T-043~T-055（path tracing/WebGPU/光学仿真/传感器噪声/PBRT）与 T-066（USD）经评估后**重组为本阶段**。目标聚焦"基于 Web 实现高精度、高质量渲染输出"，而非盲目追求全物理光学仿真。
>
> **目标**：支持 PBR 材质 + 全局光照（GI）+ 高质量 Three.js 粒子，让 Planner 产出可用于汇报/对比的高质量渲染图。
>
> **技术栈锁定**：
> - 渲染内核：Three.js / R3F（不重写）
> - 照片级 GI：`three-gpu-pathtracer` + `three-mesh-bvh`（渐进路径追踪，非 WebGPU 重写）
> - PBR 资产：glTF/GLB + KTX2 纹理（USD 仅导入交换）
> - 粒子：Three.js Points/InstancedMesh + 自研 GLSL
> - 色彩：线性工作流 + ACES/AgX + sRGB 输出
>
> **原则**：交互视口 raster 保 60fps；高质量预览才进 path tracing。

| 阶段 | 目标 | 关键任务 | 出口标准 |
|------|------|----------|----------|
| **P7 高质量渲染** | PBR + GI + 粒子 + 高质量输出 | T-080..T-090 | 见 M7 |

**执行顺序建议**：
1. **T-080** 渲染后端抽象（能力检测，不破坏现状）
2. 并行：**T-081** PBR/glTF 导入 ∥ **T-087** HDRI/IBL 资产库（互不依赖）
3. **T-082** GI 与高质量光照（依赖 T-081）
4. 并行：**T-083** Path Tracing 预览 ∥ **T-084** 粒子系统 ∥ **T-086** 后处理栈（依赖 T-082）
5. **T-085** 色调映射/色彩管理（依赖 T-083）
6. **T-088** 渲染质量预设（依赖 T-086）
7. **T-089** 相机预览 contact sheet（依赖 T-083/T-085）
8. **T-090** 渲染输出管线（依赖 T-085/T-089）= M7 验收

## 3.2 阶段3：P8 资源库与相机组工作流（当前准备进入）

> **背景**：v1 编辑器与阶段2渲染已经可用，但用户仍需要自己手动搭建相机、灯光、主体和环境。第三阶段把“资源库”作为主线：默认提供可用资源，支持常见摄像机组，一键拖入并可对组对象单独编辑。
>
> **目标**：提供一个可搜索、可预览、可下载、可复用的库系统，让用户从内置免费资源、相机阵列预设、灯光/校准/模型资产快速开始规划。
>
> **范围锁定**：
> - 本地库优先：IndexedDB + Built-in 只读库 + 用户个人库。
> - 默认资源来源必须记录 license/provenance；只使用 CC0/可再分发/官方示例等明确许可资源。
> - 常见摄像机组作为第一等资产：可拖入、可保存、可复制、可解除链接。
> - 组对象可进入“单独编辑模式”：编辑内部相机/灯光/主体，同时保留组整体 transform。
> - 远程库先做静态 catalog/download provider，不急做账号/团队同步。

| 阶段 | 目标 | 关键任务 | 出口标准 |
|------|------|----------|----------|
| **P8 资源库与相机组** | 内置免费资源 + 常见相机组 + prefab 组编辑 | T-057..T-065, T-091..T-094 | 见 M8 |

**执行顺序建议**：
1. **T-091** 免费资源 catalog 与下载器（先定义资源来源、license、缓存策略）
2. **T-064** Built-in 预设库（默认安装只读资源）
3. **T-092** 常见摄像机组预设（环绕/半球/门架/转台/线性/矩阵）
4. **T-058** Library Browser / Asset Manager UI（把 Built-in/User/Downloaded 合并进统一入口）
5. **T-059** 拖入场景 / prefab instancing（多对象保持相对 transform）
6. **T-093** 组对象单独编辑模式（进入组、隔离编辑、保存/退出）
7. **T-057** 保存选中为库资产（把用户改过的组存回个人库）
8. **T-063** 缩略图/预览渲染（让资源可识别）
9. **T-060/T-065** `.plannerlib` 导入导出与许可证来源面板
10. **T-094/T-061/T-062** 远程静态库、同步、版本冲突（P8 后半）

## 5. 里程碑与验收方法（可检验）

每个里程碑都有**客观可验证**的标准（命令/数值/截图），不靠主观判断。

### 🚩 M1 — 脚手架与核心数据（P1 出口）
**验收：**
- `npm install && npm run dev` 成功起一个含空 `<Canvas>` 的页面（附截图）。
- `npm run typecheck` 无错误。
- `npm run test` 通过：`lib/math` 角度换算、`lib/id` 前缀、`store` 基本 action 各有 ≥1 例。
- `npm run lint` 无错误。
- `src/types/` 导出 `SceneDef` 等，TS strict 下编译通过。
**怎么验：** 跑上述四命令 + 看浏览器截图 + 看 Vitest 输出。

### 🚩 M2 — 仿真核心可算（P2 出口）
**验收：**
- 给定一个**固定测试场景**（`examples/sim-fixture.json`：1 主体 box + 6 相机环绕 + 2 灯），`sim/` 输出：
  - 覆盖：主体表面最小覆盖数 > 0，盲区占比为一个确定值（单测断言该值，误差 < 1%）。
  - 重叠：6 相机两两重叠矩阵非空，平均重叠 > 0。
  - baseline：平均 baseline 在预期区间（单测断言）。
  - 曝光：方差/极差计算正确（单测断言）。
- 所有 `sim/*.test.ts` 绿。
**怎么验：** `npm run test` 看 sim 套件；`sim` 全部为纯函数，无需浏览器。

### 🚩 M3 — 可交互编辑（P3 出口）
**验收（人工 + 录屏/截图）：**
- 拖一个相机到地面 → 出现视锥线框。
- 点选 → 出现 gizmo；W/E/R 切换；拖动改变位姿，store 同步。
- Ctrl+Z 撤销 gizmo 操作可回到原位。
- Delete 删除、Ctrl+D 复制可用。
- 4 视图切换（1/2/3/4）正确。
**怎么验：** 录屏 + `store.test.ts` 对 undo/redo 的单测。

### 🚩 M4 — 专业 UI 与实时评估（P4 出口）
**验收：**
- 四区面板（大纲/视口/检视/统计）布局正确，分隔条可拖。
- Inspector 数值可键入 + scrub + 滑块三态改值，旋转用度。
- 按 Space 出覆盖热图，主体表面着色随相机增减实时变化（加相机→更蓝/绿，删相机→更红）。
- 统计条显示覆盖/盲区/重叠/baseline/曝光，超阈值告警变色。
**怎么验：** 截图对比 + 手动增减相机观察热图变化 + 阈值告警截图。

### 🚩 M5 — 导出与存盘（P5 出口）
**验收：**
- 排布一个 8 相机阵列 → 导出 `transforms.json`：
  - schema 校验通过。
  - **round-trip**：导出 → 重新解析 → 相机集与原 `SceneDef` 结构相等（位置/旋转容差 1e-6，视场角容差 1e-4）。单测断言。
- 导出 COLMAP 三件套：
  - `cameras.txt`/`images.txt` 格式正确；`points3D.txt` 为空占位。
  - 世界→相机变换正确（单测：取逆后与 transforms.json 的 camera_to_world 一致）。
- 保存 `.planner.json` → 重开加载 → 场景完全还原（单测 round-trip）。
**怎么验：** `npm run test` 看 export/io 套件；用真实 Nerfstudio/COLMAP 工具（若有）目检一次格式。

### 🚩 M6 — v1 发布候选（P6 出口）
**验收：**
- 内置示例 `examples/orbit-array.planner.json`（环绕阵列 + 主体 + 三点光）加载即用。
- 全流程冒烟：加载示例 → 调整 → 看热图 → 导出 transforms.json + 拍摄清单 CSV。
- Playwright E2E `e2e/smoke.spec.ts` 绿：打开 → 拖放一相机 → 切热图 → 导出 → 断言下载文件名与基本内容。
- `typecheck/test/lint` 全绿；README 有“如何运行”。
**怎么验：** `npm run e2e` + 手动走一遍全流程录屏。

### 🚩 M7 — 高质量渲染（P7 出口）
**验收（人工 + 截图对比）**：
- 加载一个带 PBR 材质（金属/玻璃/粗糙）+ HDRI 环境的场景，raster 视口 GI/IBL 真实感明显优于 v1（截图对比）。
- 切换 "Path Traced Preview" 模式：软阴影、反射、多次反弹 GI 渐进收敛，输出高质量截图（最高 8K）。
- 粒子系统（烟雾/火焰/尘埃至少一种）正确渲染，参数可调。
- 色彩管理：ACES/AgX 切换不爆色；导出 PNG 与视口色彩一致。
- 每相机 contact sheet 生成（缩略图网格）。
- `typecheck/test/lint` 全绿；Draft/Standard/High/Ultra 预设可切且 FPS 可接受。
**怎么验：** 截图对比（v1 vs P7）+ 渲染队列导出测试 + 性能 FPS 截图。

### 🚩 M8 — 资源库与相机组工作流（P8 出口）
**验收：**
- 首次启动后 Library 内有 Built-in 只读库，至少包含：
  - 常见摄像机组：6/12/24/48 环绕阵列、半球阵列、门架阵列、线性阵列、矩阵阵列、小物体转台。
  - 基础拍摄资源：三点布光、柔光棚、棋盘格/Charuco/AprilTag/色卡/灰卡、标准主体、若干免费模型/HDRI/材质。
- 用户可从库中搜索、预览、拖入资源；多对象资源以 group/prefab 形式实例化，保留相对位姿并自动选中新组。
- 用户可进入组对象单独编辑模式：只显示/突出当前组内容，可移动内部相机/灯光/主体；退出后组整体仍可作为一个对象移动、复制、保存。
- 免费下载资源有明确来源、license、作者、hash、下载状态；失败/取消/离线不会破坏本地库。
- 用户可把选中对象保存为个人库资产，生成缩略图，并可导出/导入 `.plannerlib` 包。
- `typecheck/test/lint` 全绿；关键库逻辑有单测，拖入/组编辑有至少一个 E2E 或可复现手动验收脚本。
**怎么验：** 清空 IndexedDB 后首次启动；检查 Built-in/Downloaded/User 三类库；拖入一个 12 相机环绕阵列并进入组编辑；保存为个人库；导出 `.plannerlib` 后重新导入。

## 6. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| `sim/` 误引入 Three 依赖 | 不可单测、难复用 | PR review + lint 规则禁止 `sim/*` import `three`/`react` |
| transforms.json 行/列主序坑 | 导出错位 | 在 `lib/math` 做唯一转置 + round-trip 单测兜底 |
| 多 coder 改 `src/types/` 冲突 | 互相破坏 | 改前 DECISION；追加可选字段；git 分支 + 小步合 |
| 热图重算卡顿 | 交互卡 | 依赖 SceneDef 哈希缓存；栅格分辨率可配 |
| 范围蔓延到光追/动态重建 | v1 永远完不成 | §1 明确排除项；超纲需求进 `docs/notes/v2.md` |
